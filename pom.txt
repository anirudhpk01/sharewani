1.Reduce or eliminate updateLookup if possible
full_document: updateLookup is expensive because it does a synchronous document read for every update event, on the critical path of event delivery. You're using it because you want the full document. But consider: your backend immediately discards the attached full document and refetches all docs for the conversation from Mongo anyway.
If that's the case, you don't need updateLookup at all. You only need the conversationId from the event, which is available in documentKey or fullDocumentBeforeChange without a lookup.
# Instead of:
collection.watch(pipeline, full_document="updateLookup")

# Use:
collection.watch(pipeline, full_document="required")  # only for inserts where full doc is already there
# or just don't set full_document for updates — you only need the conversationId from documentKey
This removes the extra read on every update event entirely. At high update volume (VOL doc being updated on every transition) this is a significant reduction in primary read load and event delivery latency.
2. Use a dedicated connection for the change stream — don't share it with your query pool
Your MongoDBConnectionPool likely serves both the change stream watcher and the batch document fetch queries that the CDC consumers run. Under heavy load, your consumers are doing large grouped queries (fetch all docs for 50 conversation IDs at once) which consume connection pool slots. If the change stream cursor is competing for connection pool slots with these queries, event delivery slows down.
Fix: maintain a separate dedicated MongoClient instance exclusively for the change stream cursor. Give it a pool size of 1 — it only needs one connection, but that connection should never be taken away by query traffic.
# Dedicated client just for CDC
cdc_client = MongoClient(
    uri,
    maxPoolSize=1,
    minPoolSize=1,
    readPreference="secondaryPreferred"
)

# Separate client for all query operations  
query_client = MongoClient(
    uri,
    maxPoolSize=50,  # sized for your consumer concurrency
    readPreference="secondaryPreferred"
)
3. Connection pool sizing for the query client
Your CDC consumers do batched Mongo reads — one grouped query per chunk, with CDC_MAX_CONCURRENCY parallel processors and CDC_NUM_CONSUMERS consumer loops. The peak number of simultaneous Mongo connections you need is roughly:
CDC_NUM_CONSUMERS × CDC_MAX_BATCHES_IN_FLIGHT × (queries per batch)
If your pool is smaller than this, consumers queue waiting for connections, which backs up the CDC queue, which causes the queue to fill, which causes silent drops. Size the query pool to comfortably exceed this number with headroom
