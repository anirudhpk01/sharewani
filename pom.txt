## How to actually fix it (pick what fits the codebase, justify your choice)

You have at least four legitimate options. Choose the one that matches the existing data model with the least disruption, and **state explicitly why you chose it**:

**Option A — Decompose the bucket document into Redis-native structures (preferred if feasible).**
Break each bucket JSON into a Redis Hash where every counter is its own field. Replace the entire read-modify-write with `HINCRBY`/`HINCRBYFLOAT` on specific fields. Distributions become hash fields per category. Timelines become sorted sets (`ZADD`/`ZINCRBY`). No JSON parsing on the write path, ever.

**Option B — Server-side mutation via Lua.**
If the JSON-blob schema must be preserved for downstream consumers, move the entire GET-mutate-SET into a single Lua script executed server-side. This collapses 3 round trips into 1 and removes Python-side JSON cost from the hot path. Still inferior to Option A but vastly better than the current state.

**Option C — Append-only delta log + lazy materialization.**
On the hot path, only `RPUSH`/`XADD` the delta to a per-bucket stream. A background materializer compacts deltas into the bucket document on a schedule or on read. Hot path becomes O(1) write, no read.

**Option D — Worker pool partitioned by bucket key.**
If contention/serialisation is part of the cost, partition buckets across N async workers by hash(bucket_key). Combine with whichever of A/B/C you pick.

I expect Option A or a hybrid of A + D unless you can justify otherwise from the code.
