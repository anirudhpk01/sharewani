**CONTEXT — Read this fully before touching any code**

You are working on a Python backend service called RealtimeAnalytics. This service listens to a MongoDB change stream (CDC) for conversation documents, builds per-conversation latest-state payloads, writes them to Redis, and maintains aggregated total-metrics buckets per market and timespan window (e.g. CUR_24H_UK, PREV_24H_UK).

**The core problem we are investigating and fixing right now:**

Across two deployed environments (DEV and SIT), both listening to CDC on the same MongoDB collection and same database, we are seeing **different total call counts** in our metrics buckets — off by thousands — with no backend crashes and no errors in logs. When data volume is low, numbers are consistent and correct. Under heavy load, they diverge. The data generation team has confirmed the same number of conversations are inserted in both cases. This is not a field-level accuracy issue. This is `total` — the most basic counter, one per conversation — being wrong. Some conversations that exist in MongoDB are simply not being counted in Redis at all.

**What we have already ruled out:**

- Backend crashes or exceptions surfacing in logs — not happening
- Data generation team inserting different data per environment — confirmed same data
- Bucketing math or field-level aggregation logic being wrong — not the focus, `total` itself is wrong
- Stale conversation state rebuilds affecting `total` — even if a conversation is rebuilt multiple times from its Mongo docs, it still contributes exactly 1 to `total`, so rebuild correctness does not affect this count

**The architectural facts you must understand before looking at any code:**

1. MongoDB CDC events trigger ingestion. Each event extracts a `conversationId` and enqueues it into an in-memory async queue.
2. A deduplication mechanism (`pending` / `dirty`) ensures a conversation already in-flight is not enqueued again — it is marked dirty and replayed once after the current processing completes.
3. Consumer workers drain the queue in batches. Each batch is split into chunks. For each chunk, all Mongo docs for all conversation IDs in that chunk are fetched in one grouped query, the latest state is rebuilt from scratch chronologically, merged with any existing Redis state, and written to both a string keyspace and a hash mirror in Redis.
4. Total metrics buckets are updated incrementally on every chunk flush (subtract old contribution, add new contribution) and also fully rebuilt from the Redis hash mirror every ~60 seconds as a healing pass.
5. The `total` counter in a bucket is simply the count of conversations in Redis whose `first_seen_at` falls within the CUR or PREV window for that market. One conversation = one count. No arithmetic involved.
6. There is no persisted CDC resume token — do not focus on that, we are not redeploying. Focus entirely on what can be fixed in the running code.
7. TTL for Redis keys is set to seconds until next midnight UTC, not a rolling 24 hours.

**What we believe is happening (hypotheses to investigate and verify in code):**

**Hypothesis 1 — Silent queue drops under load**
The CDC queue has a max size (`CDC_QUEUE_MAXSIZE`). If the watcher enqueues faster than consumers drain under heavy load, and the enqueue call is non-blocking or uses `put_nowait`, conversation IDs are silently discarded when the queue is full. No error is raised. No log is written. That conversation never reaches Redis and never contributes to `total`. Different environments with different consumer counts or queue sizes drop different numbers of IDs, producing different totals.

**Hypothesis 2 — `should_enqueue_cdc_change()` filtering differs between environments**
If this function reads any config value that differs between DEV and SIT — market filters, conversation type filters, prefix allowlists, feature flags — it will silently exclude different conversations per environment. They never enter the queue and are invisible to the entire pipeline.

**Hypothesis 3 — Silent conversation ID loss in the flush or finalize path**
If `_flush_cdc_chunk_results()` or `_finalize_cdc_chunk()` has any exception handler that catches a failure, removes the conversation ID from `pending`, but does not re-queue it — that ID is permanently lost. The code continues, no crash, no log. The conversation exists in Mongo but never in Redis.

**Hypothesis 4 — Redis hash mirror and string keyspace out of sync**
The 60s healing rebuild uses `HSCAN` on the hash mirror exclusively. If a delete pipeline succeeds on the string key but fails on `HDEL`, or if `HSET` fails while `SET` succeeds, the two representations diverge. The rebuild then counts conversations that were deleted (ghost counts) or misses conversations that were written (undercounts). Environments with different Redis memory pressure or connection pool behaviour will diverge differently.

**Hypothesis 5 — `first_seen_at` being set incorrectly, causing conversations to fall outside all bucket windows**
`total` only counts conversations whose `first_seen_at` falls within the CUR or PREV window. If `first_seen_at` is being assigned the CDC event arrival time rather than the actual document timestamp for some conversations, or if the merge logic is accidentally advancing `first_seen_at` forward instead of keeping the earliest known value, those conversations fall outside both windows and contribute 0 to `total` in all buckets. Under heavy load where doc ordering is less predictable, this would affect more conversations.

**Hypothesis 6 — 60s rebuild and incremental update racing despite the lock**
If `TOTAL_METRICS_LOCK` is acquired after the rebuild reads all hash fields but before it writes the results, an incremental update that fires in the read-to-write gap is either double-counted or lost when the rebuild overwrites the buckets. Higher CDC throughput environments (SIT) would experience more races than lower throughput ones (DEV), producing systematically different counts.

**Hypothesis 7 — TTL expiry during a long load test**
If a load test starts before midnight UTC and runs past it, keys written in the early phase expire. Those conversations vanish from Redis mid-test. The rebuild counts only surviving keys. If DEV and SIT started their load tests at different times relative to UTC midnight, their counts diverge even with identical Mongo data.
