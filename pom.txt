## Suspected remaining bottlenecks (your own caveat from last round)

You wrote:
> "total bucket JSON updates still load existing bucket documents inside `TotalMetricsManager` to preserve percentages/distributions/timelines."

This is almost certainly part of the problem. At 1400 inserts/sec + 15–20k updates/sec, a read-modify-write per bucket per event is fatal. Other likely culprits to investigate:

1. **Per-event bucket document reads** in `TotalMetricsManager` — even with pipelining, if every event triggers a GET → mutate → SET on a JSON blob, you're serialised on Redis round-trips.
2. **Synchronous / blocking I/O** anywhere on the hot path (await chains that should be `asyncio.gather`'d, sync Redis client calls, JSON (de)serialisation of large bucket docs per event).
3. **Lock contention** on shared bucket documents — concurrent updates serialising on a mutex.
4. **Single-consumer processing** — is CDC being processed by one task/worker? With 15–20k updates/sec target, you need a fan-out architecture (parallel consumers, partitioned by key).
5. **Backpressure / queue saturation** — if an internal queue between ingest and processing is bounded and the consumer is slow, ingest rate will tank to consumer rate. The 3–7/sec number smells exactly like this.
6. **Logging / observability overhead** — per-event log statements at 15k/sec will themselves bottleneck.
